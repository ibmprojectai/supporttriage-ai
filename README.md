# SupportTriage AI — Multi-Channel Support Operations Center

An AI-powered support ticket triage system built with **IBM Granite** (via OpenRouter), **LangChain**, **ChromaDB**, and **Streamlit**. Submitted to the **IBM AI Builders Challenge — July 2026** under the Wildcard theme: *Build Intelligent Systems for the Future of Work*.

---

## What it does

Most support triage tools handle email only. `supporttriage-ai` ingests tickets from **three channels simultaneously** — Email, Twitter/X, and Live Chat — and runs every ticket through a 7-stage async AI pipeline before a human ever looks at it.

**Channels:**
- **📧 Email** — formal, longer tickets with account IDs and error codes; full PII redaction before any LLM call
- **🐦 Twitter/X** — short, angry, informal messages (`@DataPilot your API is DOWN!! ERR-5001 #outage`); tone-aware classification
- **💬 Live Chat** — vague, conversational messages (`hi i cant login`, `dashboard super slow`); context-enriched before routing

**Pipeline stages (all async):**
1. **Intake** — multi-channel inbox generated by `intake/data_generator.py` (or real Zendesk stub)
2. **Guardrails** — redacts PII (emails, phones, card numbers) before any LLM call
3. **Classify** — detects category and priority; outputs a confidence score (0.0–1.0)
4. **Extract** — pulls error codes and symptoms; enables cross-channel outage detection
5. **Summarize** — condenses thread into 2–3 sentences
6. **Draft Reply** — RAG-grounded reply via metadata-filtered ChromaDB
7. **Route** — assigns queue; low-confidence tickets auto-escalate to human review

Results surface in a **Streamlit Operations Center** with three post-triage views: an executive dashboard (KPIs, Outage Radar, category + channel charts), a triaged ticket queue (expandable cards with editable drafts), and a system execution log.

## Key Innovations

| Feature | Description |
|---|---|
| **Multi-Channel Inbox** | Realistic mix of Email, Twitter/X, and Live Chat tickets — all processed by the same async pipeline. |
| **Outage Radar** | Symptoms extracted from all tickets are cross-referenced. When ≥ 2 tickets share a symptom across channels, a `🚨 POTENTIAL OUTAGE DETECTED` alert fires in real time. |
| **Confidence Scoring** | Granite outputs a 0.0–1.0 confidence score per classification. Tickets below 0.75 route to `human-review` automatically. |
| **Metadata-Filtered RAG** | ChromaDB retrieval filtered by `category` and `product` — grounded, hallucination-resistant replies. |
| **Async Batch Triage** | All 20 tickets are triaged concurrently using `asyncio.gather()` — one click, results in seconds. |
| **Severity Scoring** | Router calculates a 0.0–10.0 severity impact score from priority and critical error codes. |

## Architecture

```
intake/data_generator.py        →  generate_inbox(20) — Email + Twitter + Chat tickets
intake/zendesk_connector.py     →  fetch_ticket() / fetch_all_tickets() (Zendesk stub)
guardrails/pii_redactor.py      →  redact PII before any LLM call
pipeline/classify.py            →  category + priority + confidence_classify
pipeline/extract.py             →  error_codes + symptoms + confidence_extract
pipeline/summarize.py           →  summary
pipeline/draft.py               →  draft_reply (metadata-filtered RAG via ChromaDB)
routing/router.py               →  queue + tags + escalate + requires_human_review + severity_impact
ui/review_app.py                →  Streamlit Operations Center (Inbox → Triage → Dashboard)
watsonx_config.py               →  LLM factory: OpenRouter → Ollama → stub mode
```

## Quick Start

### 1. Clone and install

```bash
git clone https://github.com/ibmprojectai/supporttriage-ai.git
cd supporttriage-ai
pip install -r requirements.txt
```

### 2. Configure environment

```bash
cp .env.example .env
# Add your OPENROUTER_API_KEY to .env for live Granite calls
# Leave blank for stub/offline mode
```

### 3. Launch the Operations Center

```bash
streamlit run ui/review_app.py
```

Open **http://localhost:8501** — you'll see the 20-ticket inbox. Click **🚀 Run AI Triage** to watch IBM Granite process all tickets in one async batch.

## Stub / Offline Mode

If no API key is set, every LLM stage degrades gracefully — regex fallbacks and templated strings keep the pipeline running end-to-end. Confidence defaults to `0.5`, correctly triggering the human-review escalation path. `python app.py` works with zero configuration.
